## 关于问题一（每步轨迹摘要的开销与作用）

**1.1 报告的实验是否使用了每步摘要？是否尝试过增量或缓存式摘要？**

是的，报告的 Jericho 实验就是这个配置：每步用截至当前的完整轨迹生成结构化摘要，摘要文本进入动作生成的 prompt，同时也作为检索的 query（一物两用）。你对"累计摘要输入随 episode 长度近似平方增长"的判断是对的。不过在 Jericho 上这不构成实际问题：这类文字游戏每回合的文本本身非常短，即使每步都喂完整轨迹，绝对开销也很小，所以我们没有在 Jericho 上做增量优化——全量重摘要更简单，质量也更好（摘要器每次都能看到全局）。

而你提议的缓存方案实际上正是 WebArena 代码采用的：那边每步只对**新的一步**做单步摘要（`generate_single_step_summary`，缓存在 `step_summaries` 里），轨迹摘要由缓存的单步摘要拼接而成，摘要总输入是 O(T)。所以两种实现是按环境的单步文本长度选的：Jericho 单步文本短，用全量重摘要；WebArena 的 AXTree 单步就上千 token，必须用增量缓存。

**1.2 相对 base agent 的延迟/token/成本统计**

论文和 rebuttal 中报告的总成本（例如 WebArena 大预算实验的约 $200）是包含全部 LLM 调用的实测值，摘要调用已计入。但你说得对，我们此前给审稿人的每步开销分解（检索 15–47ms、advantage 估计 <0.02ms）只覆盖了检索和 advantage 计算，没有单列摘要调用——粗略地说，每步摘要相当于在动作生成之外增加约一次同量级的 LLM 调用，是 pipeline 中除动作生成外最大的开销项。我们目前没有发表过按调用类型分解的 token/延迟统计表。

**1.3 摘要收益与 advantage 修正收益的分离**

目前最接近的证据是 **Table 8**（Prompt Update vs Logit Update）：两个 arm 都保留每步摘要、使用完全相同的检索结果，只改变经验的注入通道（写进 prompt vs 修正 logits），Logit Update 仍然更好（Admin 52.31 vs 49.46，Reddit 57.64 vs 53.02）。由于摘要在两个 arm 中是公共项，这个对比能说明 logit 修正的贡献**独立于**摘要存在。

## 关于问题二（检索 pipeline 的复杂度与通用性）

**2.1 更简单的检索器会差多少？**

我们在开发阶段实际试过朴素的 embedding top-k，结论是：**主要看任务本身，而 Jericho 恰好是 embedding 检索最不擅长的那类环境**。Jericho 里大量房间的文字描述高度雷同，往往只相差几个单词，但对应的是完全不同的游戏状态和回报——这类状态在 embedding 空间里的余弦相似度都非常高，彼此几乎不可区分。纯 embedding 检索在这里的问题不是"检不到"，而是"检到的高分匹配里混着大量假阳性"，而错误匹配会把不相干的历史回报注入 advantage 估计，直接污染动作分布。这就是 pipeline 中那些词法组件存在的原因：unigram/4-gram Jaccard 重排恰好能抓住"只差几个单词"的区别（n-gram 重叠对局部词面差异敏感，embedding 反而会把它抹平），动态阈值和高相似特判也是同一目的——控制回报迁移的假阳性率。

反过来，在状态天然分离度好的环境里（不同场景的文本差异大、embedding 距离足够开），我们同意朴素的 embedding top-k 就够用，词法重排属于可省略的保护层。所以我们把这些组件理解为"针对状态空间近重复问题的保守性保护"，而不是通用必需品。补充一个相关证据：rebuttal 中的 unified pipeline 实验把两个 benchmark 的状态表示、检索流程、评估器统一成同一套后，性能与各自的环境特定版本几乎相同（Jericho 25.9→24.7，WebArena 46.98→46.01），且仍超过所有 baseline——说明增益主要来自 advantage 估计和 logit 更新机制本身。

**2.2 环境特定匹配是必要机制还是实现选择？**

我们的观点是后者。JitRL 的机制本身只要求两件事：(i) 一个对"回报可迁移性"足够可靠的状态相似度函数，(ii) 步级 reward 信号。URL 归一化这类设计只是我们发现的廉价捷径（不用额外 LLM 调用就能刻画页面类型），不是框架的组成部分——unified pipeline 实验就是直接证据。更一般的方向我们认为是用模型隐层 embedding 作为稠密状态表示，配合 ANN 检索（如 Faiss），这能把状态匹配从文本工程中解放出来，是我们的 future work。
